01 Visual Identity System
Design Verdict
The aesthetic is Precision Dark — a near-black surface system that reads like engineering tooling, not a creative portfolio. The signature element is a razor-thin violet grid in the hero that parallax-scrolls at 0.4× the page speed, suggesting depth of system — not decoration. Everything else is restrained: mono-weight type, surgical spacing, and a single high-saturation accent used exactly twice per screen.
Design constraint: The accent (#7C3AED) never appears on more than 15% of any viewport. When it appears, it means something.

02 Typography Architecture
Type Pairing
Two faces. No compromise. The goal is: display type that engineers recognize as intentional, body type that disappears into readability.
DISPLAY
--font-display Geist · 300 weight · −4% tracking
HERO WORD
--font-hero Geist · 500 weight · −2% tracking
BODY
--font-body Geist Mono · 400 · +1% tracking · used for all prose
LABEL
--font-label GEIST · 500 · CAPS · +10% TRACKING
CODE
--font-code JetBrains Mono · used for tech stack tags, metrics
Load via next/font/google with display: 'swap' and preload: true. Assign to CSS vars in the root layout. The mono-as-body choice is the single biggest signal that you think in systems.

03 Layout Philosophy
The Grid Contract
Abandon the 12-column grid as a religion. Instead: a named-area CSS grid defined per-section, where each section gets its own spatial logic. The hero is a split, the philosophy section is offset, the projects section is magazine-column. The single unifying rule: no element ever starts at the horizontal center.

04 Page-by-Page Structure
/ · Hero + Identity
app/page.tsx
Split-screen layout. Left 55%: kinetic typographic reveal of name + role using a character-stagger Framer Motion animation. Right 45%: Aceternity BackgroundBeams or GridPattern with a floating terminal-style card showing current-status metadata (timezone, open-to-work state, last-deploy timestamp pulled from GitHub API). Nav is minimal — wordmark left, 3 links right, a dot-indicator for current section. A horizontal scrolling marquee of tech stack logos sits at the viewport bottom edge.

#philosophy · Engineering Philosophy
components/PhilosophySection.tsx
A full-width manifesto statement in --text-xl Geist Light that appears via a clip-path reveal triggered by scroll intersection. Below it, three principles rendered as numbered cards (01 / 02 / 03) using Shadcn Card with a custom dark variant — each card has a hover state that reveals a short code snippet or diagram illustrating the principle. The layout is deliberately not symmetric: the manifesto statement occupies 70% width and is left-aligned, the principles grid starts at 30% and spans to full-right.
layout: offset-manifesto shadcn: Card motion: clip-reveal motion: stagger-cards
#projects · Project Deep-Dives
components/ProjectsSection.tsx + app/projects/[slug]/page.tsx
Bento grid of 4–6 projects. The featured project takes full width with a large thumbnail, 3 impact metrics (using Aceternity NumberTicker), and a "Read case study →" link. Remaining projects fill a mosaic. Each card uses Aceternity CardSpotlight — a radial gradient follows cursor position on hover. Clicking any card routes to a deep-dive page at /projects/[slug] which uses a two-column layout: problem/solution on the left, tech decisions and architecture diagrams on the right. Uses Next.js dynamic routing with generateStaticParams for all slugs.
layout: bento-mosaic aceternity: CardSpotlight, NumberTicker routing: /projects/[slug] motion: stagger-bento
#arsenal · Technical Arsenal
components/ArsenalSection.tsx
Replaces the generic skills list entirely. Uses Aceternity InfiniteMovingCards for an auto-scrolling row of technologies, each tagged with proficiency and years. Below that, a filterable grid of tools grouped by domain (Frontend, Backend, Infrastructure, Tooling). Each tool has a compact card with a custom SVG icon, proficiency bar, and one-line "used to build X" context. The filter tabs use Shadcn Tabs with a sliding indicator via Framer Motion layout animation. No star-ratings. No circular progress bars. Hard pass.

05 Animation Strategy
All animations follow three rules: (1) transform/opacity only — no layout-affecting properties, (2) scroll-triggered via Framer Motion useInView, never IntersectionObserver directly, (3) every animation respects prefers-reduced-motion via a shared useReducedMotion() hook.
01 · Hero Character Stagger Framer Motion
// Split name into characters, stagger each const container = { hidden: { opacity: 0 }, visible: { opacity: 1, transition: { staggerChildren: 0.04, delayChildren: 0.3 } } } const char = { hidden: { y: 60, opacity: 0 }, visible: { y: 0, opacity: 1, transition: { type: "spring", stiffness: 200, damping: 20 } } }
Applied to the hero h1. Each character is wrapped in a span and animated independently. Spring physics (not ease curves) give the letters weight and personality. Fire on mount, not on scroll.
02 · Scroll-Triggered Section Reveal Scroll Trigger
// Reusable hook — useReveal.ts export const revealVariants = { hidden: { opacity: 0, y: 32, filter: "blur(4px)" }, visible: { opacity: 1, y: 0, filter: "blur(0px)", transition: { duration: 0.6, ease: [0.16, 1, 0.3, 1] } } } // Usage in any section component: const ref = useRef(null) const inView = useInView(ref, { once: true, margin: "-100px" }) <motion.div ref={ref} variants={revealVariants} initial="hidden" animate={inView ? "visible" : "hidden"}>
The ease curve [0.16, 1, 0.3, 1] is a custom expo-out that feels expensive. The blur-to-sharp adds perceived depth. Use once:true so reveals don't repeat on scroll-back.
03 · Bento Grid Stagger Framer Motion
// Project cards stagger in reading order const bentoContainer = { hidden: {}, visible: { transition: { staggerChildren: 0.08 } } } const bentoItem = { hidden: { opacity: 0, scale: 0.96, y: 20 }, visible: { opacity: 1, scale: 1, y: 0, transition: { type: "spring", stiffness: 260, damping: 25 } } }
Scale from 0.96 (not 0) prevents a jarring "pop in" while still telegraphing motion. The stagger delay of 80ms feels natural without being slow.
04 · Parallax Hero Grid Framer Motion
// Background grid scrolls at 0.4× speed const { scrollY } = useScroll() const gridY = useTransform(scrollY, [0, 800], [0, -320]) const gridOpacity = useTransform(scrollY, [0, 600], [1, 0]) <motion.div style={{ y: gridY, opacity: gridOpacity }} className="absolute inset-0 pointer-events-none"> <BackgroundGrid ... /> {/* Aceternity */} </motion.div>
Fades out the grid as the user scrolls, so it doesn't compete with content below. useTransform creates a spring-less, frame-perfect transform tied to scroll position.
05 · Page Transition Framer Motion
// app/layout.tsx — wrap children in AnimatePresence // components/PageTransition.tsx export default function PageTransition({ children }) { return ( <motion.div initial={{ opacity: 0, y: 8 }} animate={{ opacity: 1, y: 0 }} exit={{ opacity: 0, y: -8 }} transition={{ duration: 0.25, ease: "easeInOut" }}> {children} </motion.div> ) }
Subtle y-offset cross-fade between routes. Fast (250ms) — page transitions that are longer than 300ms feel broken. Wrap only the main content, not the nav.
06 · Cursor Magnetic Effect (Nav + CTAs) Framer Motion
// useMagnetic.ts — attach to any button export function useMagnetic(strength = 0.4) { const x = useMotionValue(0), y = useMotionValue(0) const springX = useSpring(x, { stiffness: 300, damping: 20 }) const springY = useSpring(y, { stiffness: 300, damping: 20 }) const handleMouseMove = (e) => { const rect = e.currentTarget.getBoundingClientRect() x.set((e.clientX - rect.left - rect.width/2) * strength) y.set((e.clientY - rect.top - rect.height/2) * strength) } return { x: springX, y: springY, handleMouseMove, handleMouseLeave: () => { x.set(0); y.set(0) } } }
Apply to the primary CTA and nav links only. Not every button — magnetic effects lose impact through overuse. spring() gives the satisfying snap-back on leave.

07 Performance Architecture
High-end visual effects and sub-90 LCP are not mutually exclusive. The strategy is containment: every expensive effect is isolated in a client component, leaving the surrounding tree as static RSC.
Server-First RSC Split
Mark only Aceternity and motion components as 'use client'. All text content, metadata, and layout shells are RSC. Framer Motion and Aceternity ship zero JS to the server bundle.
Dynamic Import Boundaries
Wrap heavy Aceternity components in next/dynamic with ssr:false. BackgroundBeams and particle effects do not block the hero paint.
Image Strategy
All project thumbnails via next/image with sizes prop tuned to bento grid breakpoints. Use priority only on the hero image (if any).
CSS Containment
Apply contain: layout style paint to each section. Prevents scroll-triggered animations from triggering full-page reflows. Critical for parallax.
will-change Discipline
Only set will-change: transform during an active animation. Use Framer Motion's whileHover / whileInView — it manages this automatically.
Tailwind v4 Optimization
Use the @layer cascade correctly. Define motion variants as CSS custom properties in @theme. Avoid @apply in hot paths.
Target metrics: LCP < 1.2s on 4G, CLS = 0 (reserve space for all animated elements), INP < 100ms. Run Lighthouse CI in your GitHub Actions pipeline against every PR that touches a section component.

The Single Most Important Decision
The portfolio needs one moment that stops someone mid-scroll and makes them show a colleague. That moment is the hero character stagger combined with the parallax grid. Everything else supports it. Do not let scope creep dilute this — build the hero to 100% of spec before touching any other section. A flawless hero with an unfinished arsenal section still gets you interviews. The reverse does not.
